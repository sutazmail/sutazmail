# SutazMail

A multi-tenant, self-service mail platform on top of
[docker-mailserver](https://docker-mailserver.github.io/docker-mailserver/latest/).
One web app gives you an **admin console** (orgs, domains, mailboxes, aliases, DNS,
DKIM, settings) and a **self-service portal** where every mailbox owner manages their
own account — no shell access, no config files.

Built with Next.js 16 (App Router), React 19, TypeScript strict, Tailwind 4,
@base-ui/react + shadcn, Prisma 7 + Postgres, next-themes, sonner.

## Deploy it anywhere

```sh
git clone https://github.com/sutazmail/sutazmail.git
cd sutazmail/deploy && ./deploy.sh
```

One idempotent script deploys the full stack — mail server, REST bridge, database,
and SutazMail — generates every secret, creates the first mailbox, sets up DKIM, and
prints the DNS records to publish. See [deploy/README.md](deploy/README.md).

## What you get

**Admin console** (platform admin)

- **Dashboard** — mail-server health, live account/alias/domain counts.
- **Accounts** — create / delete mailboxes, reset passwords, one-click "Open webmail"
  per account. Creating a mailbox also provisions its portal login.
- **Aliases** — email and `/regex/` sources, multi-recipient.
- **Domains** — add a domain (tenant org + DKIM key auto-generated), publishable DNS
  records per domain with copy buttons — **fully editable**: add, change, delete, or
  repoint any record (MX, TXT, A, AAAA, CNAME, SRV, NS, CAA). Records follow global
  templates until you customize a domain; one click resets back.
- **Settings** — everything configurable from the UI, no redeploys: webmail URL,
  mail host, IMAP/SMTP ports, DNS record templates.
- **Audit log** — every mutation recorded (who, what, when).

**Self-service portal** (every mailbox owner, logging in with their own mailbox password)

- **Overview** — address, org, live storage usage.
- **Email app setup** — copyable IMAP/SMTP host + ports for any mail client.
- **Auto-responder** — vacation replies with date ranges.
- **Forwarding** — redirect with optional keep-a-copy. Pre-existing hand-made Sieve
  forwards are imported and preserved, never clobbered.
- **Security** — change own password (verified against the real mail server).

**Architecture.** The mail server stays the source of truth for mail. SutazMail adds
identity/tenancy (orgs, roles: SUPERADMIN / ORG_ADMIN / USER), sessions, audit, and
settings in Postgres. It drives docker-mailserver through two narrow channels: a
whitelisted, input-validated `setup`-CLI bridge (`src/server/mailserver.ts` — the
trust boundary) and ManageSieve (RFC 5804) with a Dovecot master account for
per-mailbox features. End users authenticate against Dovecot itself — mailbox
passwords are never stored by the app.

## Development

```sh
npm install
npm run dev        # http://localhost:3000 (needs DATABASE_URL in .env.local)
npm run typecheck
npm run test       # vitest
```

Prisma: `npx prisma migrate dev` against your local Postgres; migrations ship in
`prisma/migrations/` and are applied on container start via `prisma migrate deploy`.

## Security notes

- The REST bridge executes shell as root inside the mail container; it is reachable
  only on the internal docker network and keyed with `DMS_API_KEY`. The app builds
  every command from fixed templates with whitelisted verbs and validated,
  shell-quoted inputs (`src/server/validate.ts`).
- RBAC on every page and server action; DB-backed sessions in httpOnly signed
  cookies; login rate-limited; constant-time comparisons; TOTP-ready user model.
- Mailbox credentials are verified against Dovecot (ManageSieve STARTTLS) and never
  persisted or logged.
